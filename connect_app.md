背景
为保障前端开发能顺利对接后端功能（异常检测、MUSC交互、数据桥接等），需后端按以下规范封装接口，降低前端对接成本，确保前后端高效协作。

核心接口需求
1. 统一接口地址规范
- 需求：为每个核心功能分配固定、唯一的接口地址，所有接口添加统一版本前缀（如/api/v1/），便于前端精准调用。
- 示例格式（参考）：
  - 异常检测功能：http://localhost:5000/api/v1/anomaly/detect
  - MUSC交互功能：http://localhost:5000/api/v1/musc/interact
  - 数据桥接功能：http://localhost:5000/api/v1/data/convert
- 要求：提供所有功能的完整接口地址清单，明确每个地址对应的功能。
2. 统一请求参数格式
- 需求：明确前端传递参数的格式（统一为JSON），并指定每个功能的必填参数名、参数类型（如字符串/数组/JSON）。
- 示例格式（参考）：
  - 异常检测：前端需传 {"input_data": "用户输入的检测数据"}
  - MUSC交互：前端需传 {"musc_data": "用户输入的MUSC相关数据"}
- 要求：针对每个接口，列出“参数名+是否必填+参数类型+示例值”，确保前端传递的参数后端能正常解析。
3. 统一响应结果格式
- 需求：所有接口返回的JSON格式必须统一，便于前端快速解析和展示，避免因格式不一致增加开发成本。
- 强制统一响应模板：
{
  "code": 200, // 状态码：200=成功，400=前端参数错误，500=后端服务错误
  "msg": "操作成功", // 提示语：需为“人话”，可直接展示给用户（如“检测数据格式错误，请填写JSON格式”）
  "data": {} // 业务数据：成功时返回功能结果，失败时返回null
}
- 要求：所有接口严格按此模板返回，不随意新增/删减字段。
4. 开启跨域访问权限
- 需求：后端需配置CORS跨域支持，允许前端页面（本地开发环境/部署环境）正常调用接口，避免浏览器拦截请求。
- 要求：无需前端额外配置，后端直接处理跨域问题，确保前端本地调试时接口能正常请求。
5. 提供完整接口文档
- 需求：输出一份清晰的接口文档，包含以下内容：
  1. 所有接口的地址、请求方式（GET/POST）；
  2. 每个接口的请求参数说明（参数名、必填项、类型、示例）；
  3. 响应结果说明（各字段含义、成功/失败示例）；
  4. 常见错误码及对应的原因说明。
- 要求：文档格式简洁易懂，可直接作为前端开发的参考依据。
6. 错误提示需“用户可理解”
- 需求：接口返回的msg字段需为通俗的“人话”，避免返回仅程序员能理解的技术报错（如避免返回“SyntaxError: Unexpected token”，需返回“您输入的检测数据格式错误，请填写合法的JSON格式”）。
- 要求：所有错误提示需兼顾“前端开发调试”和“用户直接查看”，无需前端二次翻译。
期望交付物
1. 可运行的后端服务（确保接口能正常访问）；
2. 完整的接口文档（含示例）；
3. 错误码对照表（可选，补充说明不同code对应的具体原因）。
